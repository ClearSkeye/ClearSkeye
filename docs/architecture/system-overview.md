# System Architecture Overview

## 1. Application profile

ClearSkeye is a single-repository web application with two runtime planes:

1. **Browser SPA runtime** (React + Vite output) served as static assets.
2. **Serverless API runtime** (Vercel Node functions) served from `api/*.ts`.

The project is intentionally minimal:

- One page (`src/App.tsx`) and no client route table.
- Two API endpoints (`/api/contact`, `/api/ghost-posts`).
- No internal database.
- No queue workers or cron jobs.
- External service dependencies for email and blog content.

---

## 2. Deployment topology

### 2.1 Hosting model

- **Platform:** Vercel
- **Frontend delivery:** static assets generated by Vite.
- **Backend delivery:** Vercel serverless functions from files in `api/`.

### 2.2 Routing model

- `vercel.json` rewrites all paths to `/index.html`:
  - Source: `/(.*)`
  - Destination: `/index.html`
- API routes are still resolved by Vercel's function routing (`/api/*`) and are not consumed by the SPA fallback.

### 2.3 External dependencies by runtime

- **Browser runtime**
  - Ghost Portal script (`members.js`) loaded from `VITE_GHOST_URL` when configured.
  - Vercel Analytics + Speed Insights client integrations.
- **Serverless runtime**
  - Resend API (`POST /api/contact`).
  - Ghost Content API (`GET /api/ghost-posts`).

## 3. End-to-end runtime flows

## 3.1 Initial page load

1. Browser requests site root.
2. Vercel serves `index.html`.
3. Browser loads `/src/main.tsx` bundle output.
4. `src/main.tsx` mounts React app with:
   - `StrictMode`
   - `HelmetProvider`
   - `<Analytics />`
   - `<SpeedInsights />`
5. `src/App.tsx` renders page sections and utility components (`PageSeo`, `GhostPortalScript`, `Nav`, `Footer`).
6. If `VITE_GHOST_URL` exists, `GhostPortalScript` appends a `<script>` tag to load Ghost Portal from `<ghost-url>/members.js`.

## 3.2 Contact submission

1. User submits form in `src/components/sections/Contact.tsx`.
2. `submitContact()` in `src/lib/contact.ts` validates payload via `ContactRequestSchema`.
3. Browser sends `POST /api/contact` JSON body.
4. `api/contact.ts` re-validates payload via server-side `ContactSchema`.
5. If honeypot field is populated, endpoint returns success (`200`, `{ ok: true }`) and exits.
6. Endpoint reads `RESEND_API_KEY`, `CONTACT_TO_EMAIL`, and `CONTACT_FROM_EMAIL`.
7. Endpoint sends outbound email via `resend.emails.send(...)`.
8. Endpoint responds with `200 { ok: true }` on success.
9. Contact component transitions local status state (`submitting -> success` or `submitting -> error`).

## 3.3 Latest blog posts

1. `LatestPosts` mounts and runs `fetchLatestGhostPosts()` once.
2. Browser calls `GET /api/ghost-posts`.
3. `api/ghost-posts.ts` verifies `GHOST_URL` and `GHOST_CONTENT_API_KEY`.
4. Endpoint requests Ghost Content API:
   - Path: `/ghost/api/content/posts/`
   - Query: `limit=3`, `fields=id,title,url,excerpt,published_at`, `formats=plaintext`
5. Endpoint validates response with `GhostPostsResponseSchema`.
6. Endpoint returns normalized JSON payload `{ posts: [...] }`.
7. Frontend renders cards if posts exist; section is hidden when response is unavailable/empty.

## 3.4 Newsletter CTA behavior

1. User clicks subscribe button in `NewsletterCta`.
2. `openGhostPortal()` in `src/lib/ghost.ts` attempts:
   - `window.Portal.open()` if Ghost script initialized.
   - Fallback redirect to `${VITE_GHOST_URL}/#/portal/signup` if `window.Portal` is absent and URL is configured.
3. If `VITE_GHOST_URL` is missing, button intentionally performs no navigation.

## 4. Module boundaries and responsibilities

### 4.1 Browser module boundary (`src/`)

- **Composition layer:** `src/App.tsx`
- **Section components:** `src/components/sections/*`
- **Global utility components:** SEO and script injectors in `src/components/*`
- **Runtime helpers:** `src/lib/*`
- **Static content config:** `src/content/site.ts`

No state management library is used; local `useState` and `useEffect` drive interaction and async flows.

### 4.2 API module boundary (`api/`)

- Endpoint-per-file style.
- No middleware chain.
- Each endpoint performs:
  1. method gating
  2. input/env validation
  3. external request handling
  4. response serialization

### 4.3 Config/tooling boundary (repo root)

- Build and dev orchestration in `package.json`.
- Bundler and CSS toolchain via `vite.config.ts` and `tailwind.config.ts`.
- TypeScript split configs (`tsconfig.app.json`, `tsconfig.node.json`) referenced by `tsconfig.json`.
- Lint config in `eslint.config.js`.
- Deployment routing in `vercel.json`.

## 6. Architectural constraints

1. **Single-page IA:** all navigation is in-page anchor scrolling.
2. **No persistence layer:** the app itself stores no records or user content.
3. **External-service reliance:** contact delivery and latest posts depend on third-party availability.
4. **Client + server validation duplication:** both layers parse/validate contact payload shape.
5. **No shared backend framework:** endpoint logic is duplicated by design for this project size.

---

## 7. Scaling and extension points

If architecture evolves, the current clean seams are:

- Add new APIs under `api/` and corresponding client helpers in `src/lib/`.
- Introduce shared backend helpers in `api/_shared/` if endpoint count grows.
- Move from in-page sections to route-based navigation if product depth increases.
- Add persistent storage only when data retention requirements appear.

Any such change should be reflected in:

- `docs/architecture/*`
- `docs/configuration/*`
- `docs/operations/deployment-and-runbooks.md`
